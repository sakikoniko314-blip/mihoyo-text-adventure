这是一个原神崩铁的文字冒险游戏。整个游戏基于由游戏内文本构建的完整文本库。这使得 AI 能够完整地获取到相当全面的世界观和角色设定。选择世界和起始地点，AI 续写故事，你选选项推进。

文本库

从 Genshin Impact 和 Honkai Star Rail 的游戏内文本中提取了角色介绍、语音、书籍、物品描述、任务对话等内容，构建 FTS5 全文索引。AI 生成叙事时，根据当前上下文自动检索相关设定并注入 prompt，确保角色行为和世界观引用准确——俗称 RAG。

数据规模：117 个原神角色 + 83 个崩铁角色的人设摘要，以及对应的地点和阵营信息。

双 AI 架构

整个流程分两次 DeepSeek 调用：

1. **叙事 AI** — 接收游戏状态、历史记录、RAG 检索结果，产出叙事段落和 `【故事状态】`（地点/在场角色/当前目标）。
2. **选项 AI** — 基于叙事和当前状态，生成 4 个具体行动选项。选项 prompt 要求禁止空洞动词（"观察""继续"等），强制产出衔接剧情的具体动作。

拆开的好处：叙事质量不受选项格式约束影响，选项也不会因为 token 被叙事挤占而缺失。

剧情终点

叙事 AI 在判断剧情达到自然章节终点时输出 `【冒险结束】` 而非 `【故事状态】`。服务端检测到后在前端注入金色的"迎接冒险终章"选项。玩家选择后触发独立终章调用，生成收束叙事并展示结局画面。避免故事永远在循环。

上下文管理

每轮对话的 `【故事状态】` 在前后端之间传递，维持连续性。历史对话每约 12 轮触发一次 LLM 摘要压缩，摘要内容插入 history 作为 `summary_marker`，后续请求只取摘要之后的最近几轮。目的是在 token 窗口内保持可接受的故事记忆。

前端

单个 HTML 文件，零外部依赖。打字机逐字渲染、Canvas 粒子背景、轮次时间线可跳转。深色主题，移动端适配。存档读档通过 localStorage 实现。
